# Kubernetes Notes

# 1. Deployment vs StatefulSet

| Feature | Deployment | StatefulSet |
|----------|------------|-------------|
| Application Type | Stateless | Stateful |
| Examples | Nginx, Spring Boot, React | MySQL, PostgreSQL, Kafka, MongoDB |
| Pod Identity | Random | Stable (`pod-0`, `pod-1`) |
| Storage | Shared/Ephemeral | Dedicated Persistent Volume per Pod |
| DNS | Dynamic | Stable |
| Scaling | Any order | Sequential |
| Rolling Update | Any order | Ordered |
| Pod Replacement | New pod with new identity | Same pod name and storage |
| Service Used | ClusterIP / LoadBalancer | Headless Service |

---

## Deployment Architecture

```
              kubectl apply
                     │
                     ▼
              API Server
                     │
                     ▼
                   etcd
                     │
                     ▼
         Deployment Controller
                     │
                     ▼
               ReplicaSet
                     │
         ┌───────────┼───────────┐
         ▼           ▼           ▼
      Pod-A       Pod-B       Pod-C
```

### Background Working

1. User runs

```bash
kubectl apply -f deployment.yaml
```

2. API Server stores Deployment in etcd.

3. Deployment Controller detects the Deployment.

4. Controller creates a ReplicaSet.

5. ReplicaSet creates Pods.

6. If a Pod dies

```
Pod-B ❌

↓

New Pod-X ✅
```

The new Pod gets:
- New name
- New IP
- Fresh identity

Deployment only cares about the number of replicas, not which specific Pod exists.

---

## StatefulSet Architecture

```
               kubectl apply
                      │
                      ▼
                API Server
                      │
                      ▼
                    etcd
                      │
                      ▼
          StatefulSet Controller
                      │
        ┌─────────────┴─────────────┐
        ▼                           ▼
    mysql-0                    PVC-0
        │
      Ready
        │
        ▼
    mysql-1                    PVC-1
        │
      Ready
        │
        ▼
    mysql-2                    PVC-2
```

### Background Working

1. User applies StatefulSet.

2. API Server stores it in etcd.

3. StatefulSet Controller watches the API Server.

4. Creates Pods one at a time.

```
mysql-0

↓

Ready

↓

mysql-1

↓

Ready

↓

mysql-2
```

5. Each Pod gets

- Stable hostname
- Stable DNS
- Dedicated PVC

Example

```
mysql-0
     │
PVC-mysql-0
     │
Persistent Volume
```

---

### If a Pod crashes

```
mysql-1 ❌
```

StatefulSet recreates

```
mysql-1 ✅
```

NOT

```
mysql-8
```

The recreated Pod gets

- Same name
- Same DNS
- Same PVC
- Same data

---

# Why StatefulSet uses Headless Service

Stateful applications need to talk to a **specific Pod**.

Example

```
Primary Database

↓

mysql-0

Replica

↓

mysql-1
```

Replica must always connect to **mysql-0**.

A normal Service cannot guarantee this because it load balances.

---

# 2. Headless Service

A Headless Service is simply

```yaml
clusterIP: None
```

Example

```yaml
apiVersion: v1
kind: Service

metadata:
  name: mysql

spec:
  clusterIP: None

  selector:
    app: mysql

  ports:
  - port: 3306
```

---

## Normal ClusterIP Service

```
Application
      │
      ▼
Service IP
10.96.0.10
      │
 ┌────┼────┐
 ▼    ▼    ▼
Pod1 Pod2 Pod3
```

Traffic is load balanced.

Application doesn't know which Pod it reaches.

---

## Headless Service

```
Application
      │
      ▼
DNS
      │
 ┌────┼────┐
 ▼    ▼    ▼
Pod1 Pod2 Pod3
```

No ClusterIP.

No load balancing.

DNS returns Pod IPs directly.

---

## DNS Entries Created

For StatefulSet

```
mysql-0
mysql-1
mysql-2
```

Headless Service creates

```
mysql-0.mysql.default.svc.cluster.local

mysql-1.mysql.default.svc.cluster.local

mysql-2.mysql.default.svc.cluster.local
```

Applications can connect directly.

Example

```
mysql-1

↓

mysql-0.mysql.default.svc.cluster.local
```

Always reaches mysql-0.

---

# Background Working of Headless Service

When Kubernetes sees

```yaml
clusterIP: None
```

it does **not** allocate a virtual IP.

Instead it registers DNS records.

```
CoreDNS

mysql

│

├── mysql-0 → 10.0.0.5

├── mysql-1 → 10.0.0.6

└── mysql-2 → 10.0.0.7
```

When an application performs DNS lookup

```
mysql.default.svc.cluster.local
```

CoreDNS returns Pod IPs.

Application connects directly to Pods.

# Deployment vs StatefulSet Flow

## Deployment

```
Deployment

↓

ReplicaSet

↓

Pods

↓

Service

↓

Client
```

Pods are interchangeable.

---

## StatefulSet

```
StatefulSet

↓

mysql-0
mysql-1
mysql-2

↓

Headless Service

↓

Stable DNS

↓

Application
```

Each Pod has a fixed identity.

---

# Interview Summary

## Deployment

- Used for stateless applications.
- Creates ReplicaSet.
- Pods have random names.
- Pods are interchangeable.
- Usually used with ClusterIP or LoadBalancer Service.

## StatefulSet

- Used for stateful applications.
- Pods have stable names.
- One Persistent Volume per Pod.
- Ordered creation and deletion.
- Uses Headless Service.

## Headless Service

- `clusterIP: None`
- No virtual IP.
- No load balancing.
- Returns Pod IPs through DNS.
- Required when applications must communicate with specific Pods.
