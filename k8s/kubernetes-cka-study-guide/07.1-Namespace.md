# Kubernetes Namespace

## What is a Namespace?

A **Namespace** is a logical partition inside a Kubernetes cluster that allows you to organize, isolate, and manage resources.

Think of a Namespace as a **virtual cluster** inside a single Kubernetes cluster.

```
Kubernetes Cluster
│
├── default
│     ├── Pod A
│     ├── Service A
│
├── dev
│     ├── Pod B
│     ├── Deployment B
│
├── test
│     ├── Pod C
│
└── prod
      ├── Pod D
      ├── Service D
      └── ConfigMap
```

Namespaces do **not** create separate clusters. They only separate resources logically.

---

# Why Do We Need Namespaces?

Imagine a Kubernetes cluster used by multiple teams.

Without namespaces:

```
Cluster

Frontend Team
Backend Team
DevOps Team
QA Team

All resources mixed together
```

Problems:

- Name conflicts
- Difficult resource management
- Difficult access control
- Difficult monitoring
- Difficult resource quotas

Namespaces solve these problems.

---

# Real-World Example

Suppose a company has three environments.

```
Production
Testing
Development
```

Instead of creating three Kubernetes clusters, create three namespaces.

```
Cluster

├── dev
├── test
└── prod
```

Each team works inside its own namespace.

---

# Default Namespaces

Kubernetes creates some namespaces automatically.

| Namespace | Purpose |
|------------|---------|
| default | User applications |
| kube-system | Kubernetes system components |
| kube-public | Publicly readable resources |
| kube-node-lease | Node heartbeat information |

View namespaces:

```bash
kubectl get namespaces
```

Output

```
NAME              STATUS   AGE
default           Active
kube-system       Active
kube-public       Active
kube-node-lease   Active
```

# How Namespace Works in the Background

When you create a resource like a Pod:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  namespace: dev
```

The request flow is:

```
kubectl apply
       │
       ▼
API Server
       │
Stores metadata:
Name = nginx
Namespace = dev
UID
Labels
Annotations
       │
       ▼
etcd
       │
       ▼
Scheduler
       │
Schedules Pod to Node
       │
       ▼
Kubelet creates Pod
```

The namespace is stored as metadata in the Kubernetes API and in **etcd**.

---

# Does Namespace Create Separate Nodes?

No.

Example

```
Node-1
 ├── dev/nginx
 ├── test/mysql
 └── prod/api

Node-2
 ├── dev/redis
 └── prod/nginx
```

Pods from different namespaces can run on the same node.

Namespace provides logical isolation, not physical isolation.

---

# Resource Name Uniqueness

Inside one namespace:

```
default

nginx
nginx
```

❌ Not allowed

Different namespaces:

```
dev/nginx

test/nginx

prod/nginx
```

✅ Allowed

# Create Namespace

Using command

```bash
kubectl create namespace dev
```

Verify

```bash
kubectl get ns
```

---

# Create Namespace Using YAML

namespace.yaml

```yaml
apiVersion: v1
kind: Namespace

metadata:
  name: dev
```

Apply

```bash
kubectl apply -f namespace.yaml
```

---

# Create Pod Inside Namespace

pod.yaml

```yaml
apiVersion: v1
kind: Pod

metadata:
  name: nginx

  namespace: dev

spec:
  containers:
  - name: nginx
    image: nginx
```

Apply

```bash
kubectl apply -f pod.yaml
```

Verify

```bash
kubectl get pods -n dev
```

---

# Using kubectl Without Modifying YAML

Instead of adding

```yaml
namespace: dev
```

You can run

```bash
kubectl apply -f pod.yaml -n dev
```

or

```bash
kubectl create -f pod.yaml -n dev
```

**Note:** If the YAML already specifies `metadata.namespace`, that value takes precedence. The `-n` flag is mainly used when the manifest does not define a namespace.

---

# Set Default Namespace

Current namespace

```bash
kubectl config view --minify
```

Set namespace

```bash
kubectl config set-context --current --namespace=dev
```

Now every command uses

```
dev
```

unless another namespace is specified.

---

# List Resources From Namespace

Pods

```bash
kubectl get pods -n dev
```

Deployments

```bash
kubectl get deploy -n dev
```

Services

```bash
kubectl get svc -n dev
```

Everything

```bash
kubectl get all -n dev
```

---

# Delete Namespace

```bash
kubectl delete namespace dev
```

Everything inside that namespace is deleted.

---

# Benefits of Namespace

- Logical isolation
- Resource organization
- Same resource names in different namespaces
- Team separation
- Environment separation (Dev/Test/Prod)
- Resource quotas
- RBAC (Role-Based Access Control)
- Easier monitoring and management

---

# Limitations

Namespaces do **not**:

- Create separate clusters
- Create separate nodes
- Provide complete network isolation by themselves (use **NetworkPolicies** if required)
- Provide separate etcd databases

---

# Best Use Cases

Use namespaces for:

- Development environment
- Testing environment

- Multiple applications
- Multi-tenant clusters

# Interview Questions

## 1. What is a Namespace?

A Namespace is a logical partition within a Kubernetes cluster used to organize and isolate resources.

---

## 2. Why do we use Namespaces?

- Avoid resource name conflicts
- Separate teams and environments
- Apply RBAC
- Apply ResourceQuotas
- Organize cluster resources

---

## 3. Are Namespaces physically isolated?

No.

They provide logical isolation only.

---

## 4. Can two Pods have the same name?

Yes, if they are in different namespaces.

Example

```
dev/nginx

prod/nginx
```

---

## 5. Can Pods from different namespaces communicate?

Yes, by default they can communicate if networking allows it. Namespaces alone do not block traffic. To restrict communication, use **NetworkPolicies**.

---

## 6. Where is Namespace information stored?

Namespace information is stored as metadata in the Kubernetes API and persisted in **etcd**.

---

# Summary

- Namespace is a logical boundary inside a Kubernetes cluster.
- It helps organize resources and separate teams or environments.
- Resources in different namespaces can have the same name.
- Namespaces are stored in etcd and managed by the API Server.
- Add `metadata.namespace` to a resource manifest or use `kubectl -n` to deploy into a specific namespace.
- Common namespaces include `default`, `kube-system`, `kube-public`, and `kube-node-lease`.
