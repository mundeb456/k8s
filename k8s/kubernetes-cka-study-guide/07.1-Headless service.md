# Kubernetes Headless Service

## What is a Headless Service?

A **Headless Service** is a Kubernetes Service that **does not receive a Cluster IP**. Instead of routing traffic through a virtual IP and load-balancing requests, it exposes the **individual IP addresses of the Pods** behind the Service.

A Headless Service is created by setting:

```yaml
clusterIP: None
```

---

## Why Do We Need a Headless Service?

Normally, Kubernetes Services provide:

- A stable virtual IP (ClusterIP)
- Load balancing across Pods

However, some applications need to communicate with **specific Pods** instead of any available Pod.

Examples:
- MySQL Cluster
- MongoDB Replica Set
- Kafka
- Cassandra
- ZooKeeper
- Elasticsearch

These applications require each Pod to have its own identity.

---

## How a Normal ClusterIP Service Works

```
               Client
                  |
                  |
          +----------------+
          | ClusterIP      |
          | Service        |
          +----------------+
                  |
         ---------------------
         |         |         |
      Pod-1     Pod-2     Pod-3
```

### Flow

1. Client sends request to Service.
2. Service receives the request.
3. kube-proxy performs load balancing.
4. One Pod receives the request.

The client never knows which Pod handled the request.

---

## How a Headless Service Works

```
               Client
                  |
          DNS Lookup
                  |
        ---------------------
        |        |         |
      Pod-1    Pod-2     Pod-3
```

### Flow

1. Client queries DNS.
2. DNS returns the IP addresses of all matching Pods.
3. Client chooses which Pod to communicate with.
4. No Kubernetes load balancing occurs.

---

## Headless Service YAML

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-headless-service
spec:
  clusterIP: None
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
```

---

## DNS Resolution

Suppose we have:

- Service Name: `mysql`
- Namespace: `default`

DNS Name:

```
mysql.default.svc.cluster.local
```

Instead of returning one ClusterIP, DNS returns:

```
10.244.1.5
10.244.1.6
10.244.1.7
```

Each IP belongs to a different Pod.

---

## StatefulSet Example

Pods in a StatefulSet receive stable names.

Example:

```
mysql-0
mysql-1
mysql-2
```

With a Headless Service, each Pod gets a stable DNS name.

```
mysql-0.mysql.default.svc.cluster.local

mysql-1.mysql.default.svc.cluster.local

mysql-2.mysql.default.svc.cluster.local
```

Applications can directly communicate with a particular Pod.

---

## ClusterIP vs Headless Service

| Feature | ClusterIP | Headless Service |
|----------|-----------|------------------|
| Cluster IP | Yes | No |
| Load Balancing | Yes | No |
| DNS Returns | Service IP | Individual Pod IPs |
| Client Connects To | Service | Pod |
| kube-proxy | Used | Not Used |
| Best For | Stateless Applications | Stateful Applications |

---

## Advantages

- Direct Pod communication
- Stable Pod identities
- Required for StatefulSets
- Better service discovery
- No load-balancing overhead

---

## Disadvantages

- Client performs Pod selection.
- No built-in load balancing.
- Slightly more complex client logic.

---

## When to Use Headless Services

Use Headless Services when:

- Using StatefulSets
- Running databases
- Running Kafka clusters
- Running Cassandra
- Running ZooKeeper
- Applications require direct Pod-to-Pod communication
- Every Pod has a unique role

---

## When NOT to Use

Avoid Headless Services when:

- Building stateless web applications
- Running REST APIs
- Deploying frontend applications
- Kubernetes load balancing is sufficient

Use a normal **ClusterIP** Service instead.

# Interview Questions

### 1. What is a Headless Service?

A Headless Service is a Kubernetes Service created with `clusterIP: None`. It does not allocate a Cluster IP or perform load balancing. Instead, DNS returns the IP addresses of individual Pods.

---

### 2. Why do we use a Headless Service?

To allow applications to communicate directly with individual Pods, especially in StatefulSets where each Pod has a unique identity.

---

### 3. Does a Headless Service perform load balancing?

No.

The client decides which Pod to connect to.

---

### 4. What does DNS return for a Headless Service?

DNS returns the IP addresses of all matching Pods instead of a single Service IP.

---

### 5. Which Kubernetes object commonly uses a Headless Service?

**StatefulSet**

---

### 6. What is the difference between ClusterIP and Headless Service?

| ClusterIP | Headless |
|------------|-----------|
| Has Virtual IP | No Virtual IP |
| Load Balancing | No Load Balancing |
| Used for Stateless Apps | Used for Stateful Apps |

---

## Key Takeaways

- `clusterIP: None` creates a Headless Service.
- No virtual IP is assigned.
- No kube-proxy load balancing.
- DNS returns individual Pod IPs.
- Commonly used with StatefulSets.
- Ideal for databases and clustered applications.
