# Kubernetes PriorityClass – Detailed Explanation

## What is a PriorityClass?

A **PriorityClass** determines the **importance (priority)** of a Pod relative to other Pods in the cluster.

Kubernetes uses pod priority to decide:

1. Which pending Pod should be scheduled first when resources are limited.
2. Which running Pods should be evicted (preempted) to make room for a higher-priority Pod.

In simple terms:

- Higher priority Pods are scheduled before lower priority Pods.
- Higher priority Pods can preempt lower priority Pods if necessary.

---

# Why do we need PriorityClass?

Imagine a Kubernetes cluster with limited resources.

Applications running in the cluster:

- Database
- API Server
- Monitoring
- Batch Jobs
- Analytics Jobs

If all Pods have the same priority, Kubernetes treats them equally.

However, in production:

- Database must always run.
- API Server must always run.
- Monitoring should remain available.
- Batch jobs can wait.

PriorityClass enables Kubernetes to automatically make these decisions.

---

# How PriorityClass Works

A PriorityClass is a cluster-wide resource.

Example:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 100000
globalDefault: false
description: "Critical applications"
```

Here:

```
PriorityClass Name : high-priority

Priority Value : 100000
```

Pods use this PriorityClass by specifying:

```yaml
spec:
  priorityClassName: high-priority
```

---

# Priority Value

The most important field is:

```yaml
value:
```

Higher value means higher priority.

Example:

| PriorityClass | Value |
|--------------|------:|
| low | 100 |
| medium | 1000 |
| high | 10000 |
| critical | 100000 |

---

# Example Cluster

Suppose a node has:

```
CPU    : 8 Cores
Memory : 16 GB
```

Running Pods:

| Pod | CPU | Priority |
|------|----:|---------:|
| Batch Job | 2 | 100 |
| Analytics | 2 | 100 |
| Test App | 2 | 100 |
| Jenkins | 2 | 100 |

Total CPU used:

```
2 + 2 + 2 + 2 = 8 CPUs
```

The node is full.

Now a new Pod arrives.

```
Pod: Database

CPU Required = 2

Priority = 100000
```

No CPU is available.

Scheduler compares priorities.

```
Database : 100000

Others : 100
```

Since the Database Pod has a higher priority, Kubernetes evicts one low-priority Pod.

Result:

Before:

```
Batch
Analytics
Test
Jenkins
```

After:

```
Database
Analytics
Test
Jenkins
```

The Database Pod gets scheduled.

This process is called **Preemption**.

# Creating a PriorityClass

Low Priority:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: low-priority
value: 100
globalDefault: false
description: "Low priority workloads"
```

Medium Priority:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: medium-priority
value: 1000
globalDefault: false
description: "Medium priority workloads"
```

High Priority:

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 100000
globalDefault: false
description: "Production applications"
```

---

# Using PriorityClass in a Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  replicas: 2
  selector:
    matchLabels:
      app: payment
  template:
    metadata:
      labels:
        app: payment
    spec:
      priorityClassName: high-priority
      containers:
      - name: payment
        image: nginx
```

All Pods created by this Deployment will have high priority.

# Scheduling Order Example

Pending Pods:

| Pod | Priority |
|------|---------:|
| Batch | 100 |
| Database | 100000 |
| Logging | 500 |
| API | 50000 |

Scheduler order:

```
1. Database
2. API
3. Logging
4. Batch
```

Even if Batch arrived first, Database is scheduled before it because it has a higher priority.

---

# Without PriorityClass

Arrival order:

```
Pod A

Pod B

Pod C

Pod D
```

Scheduler processes them in the same order:

```
A

B

C

D
```

---

# With PriorityClass

Arrival order:

```
Batch

API

Database

Logging
```

Priorities:

```
Batch     = 100

API       = 50000

Database  = 100000

Logging   = 1000
```

Scheduler order:

```
Database

API

Logging

Batch
```

Priority takes precedence over arrival time.

---

# Preemption Example

Node Capacity:

```
CPU = 4
```

Running Pods:

| Pod | CPU | Priority |
|------|----:|---------:|
| Batch | 2 | 100 |
| Analytics | 2 | 100 |

The node is full.

A new Pod arrives:

```
Database

CPU = 2

Priority = 100000
```

Scheduler cannot fit the Pod.

It preempts one lower-priority Pod.

Result:

```
Analytics

Database
```

---

# What if no lower-priority Pod exists?

Running Pods:

| Pod | Priority |
|------|---------:|
| API | 100000 |
| Database | 100000 |

Incoming Pod:

```
Priority = 50000
```

Since every running Pod has a higher priority, Kubernetes cannot evict them.

The new Pod remains in the **Pending** state until resources become available.

---

# globalDefault

Example:

```yaml
globalDefault: true
```

If a Pod does not specify:

```yaml
priorityClassName:
```

it automatically receives the default PriorityClass.

Only **one** PriorityClass in a cluster can have `globalDefault: true`.

---

# Priority vs Resource Requests

Priority does **not** reserve CPU or Memory.

Example:

```
High Priority Pod

CPU Request = 8

Available CPU = 2
```

Even with the highest priority, the Pod cannot be scheduled unless enough resources become available.

Priority determines **which Pod gets scheduled first**, not **whether resources exist**.

---

# Priority vs Node Affinity

Priority answers:

> Which Pod should run first?

Node Affinity answers:

> Which node should the Pod run on?

These are different concepts.

---

# Priority vs QoS

Quality of Service (QoS) classes determine eviction behavior when a node experiences memory or CPU pressure.

QoS Classes:

- Guaranteed
- Burstable
- BestEffort

PriorityClass determines scheduling order and preemption.

A Pod can have:

- High Priority
- BestEffort QoS

at the same time.

---

# Built-in PriorityClasses

Kubernetes provides two reserved PriorityClasses:

```
system-node-critical

system-cluster-critical
```

These are used for critical Kubernetes components such as:

- CoreDNS
- kube-proxy
- Networking plugins
- Control Plane components

Application workloads should generally not use these classes.

---

# Best Practices

- Use **High Priority** for:
  - Databases
  - API Servers
  - Ingress Controllers
  - Monitoring

- Use **Medium Priority** for:
  - Business applications
  - Backend services

- Use **Low Priority** for:
  - Batch Jobs
  - CI/CD
  - Analytics
  - Development workloads

Avoid assigning every workload a high priority. If everything has the highest priority, Kubernetes cannot distinguish which workloads are truly critical.

# Complete Flow Diagram

```
                     New Pod
                        │
                        ▼
             Does node have resources?
                 │               │
              YES                NO
               │                 │
               ▼                 ▼
      Schedule Pod      Check Pod Priority
                                │
                                ▼
               Higher priority than running Pods?
                     │                 │
                  YES                 NO
                   │                  │
                   ▼                  ▼
        Evict Lower Priority     Pod stays Pending
             Pod(s)
                   │
                   ▼
          Schedule New Pod
```

# Summary

| Feature | Description |
|----------|-------------|
| PriorityClass | Defines the scheduling priority for Pods |
| value | Numeric priority value; higher values indicate higher priority |
| priorityClassName | Assigns a PriorityClass to a Pod |
| Scheduling | Higher-priority Pods are scheduled before lower-priority Pods when resources are constrained |
| Preemption | Higher-priority Pods can evict lower-priority Pods if needed |
| globalDefault | Default PriorityClass for Pods without one specified |
| Built-in Classes | `system-node-critical` and `system-cluster-critical` for Kubernetes system components |
